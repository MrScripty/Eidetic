# src-tauri/gen/schemas

## Purpose

This directory contains generated JSON schemas for Tauri configuration and
capability manifests.

## Contents

| File | Description |
| ---- | ----------- |
| `desktop-schema.json` | Desktop configuration schema. |
| `linux-schema.json` | Linux platform configuration schema. |
| `capabilities.json` | Capability manifest schema. |
| `acl-manifests.json` | Access-control manifest schema. |

## Problem

Schema artifacts need to be discoverable without being mistaken for manually
owned application logic.

## Constraints

- The files are generated by Tauri tooling.
- Manual edits may be overwritten.
- Capability and configuration validation depends on schema compatibility.

## Decision

Keep generated schemas in place and treat updates as tooling/dependency
changes.

## Alternatives Rejected

- Hand-maintaining schemas: rejected because Tauri owns schema generation.
- Removing schemas: rejected because tooling and review workflows may rely on
  them.

## Invariants

- Schema files stay valid JSON.
- Application policy is expressed in source code and capability manifests, not
  by editing generated schemas.

## Revisit Triggers

- Tauri schema generation changes.
- CI begins checking generated artifact drift.

## Dependencies

**Internal:** `src-tauri/capabilities`.
**External:** Tauri schema generation.

## Related ADRs

- `ADR-002` standards compliance baseline.

## Usage Examples

```bash
jq '.title' src-tauri/gen/schemas/capabilities.json
```

## API Consumer Contract

- None for runtime API consumers.
- Reason: schemas are tooling artifacts.
- Revisit trigger: external tooling consumes these schemas directly.

## Structured Producer Contract

- JSON schemas are structured artifacts produced by Tauri tooling and consumed
  by validation/editor integrations.
